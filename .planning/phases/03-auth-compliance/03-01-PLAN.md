**In `apps/api/src/routes/auth.ts`:**

Add a new exported function `requireV1ApiPrincipal` after the existing `requireApiPrincipal` function. This function is bearer-only auth for `/v1/*` routes. It does NOT call `resolvePrincipal` — it has its own minimal resolution logic:

```typescript
export async function requireV1ApiPrincipal(
  request: FastifyRequest,
  reply: FastifyReply,
  services: RuntimeServices,
  requiredScope: "chat" | "image" | "usage" | "billing",
): Promise<AuthPrincipal | undefined> {
  const bearerToken = readBearerToken(request);
  if (!bearerToken) {
    sendApiError(reply, 401, "No API key provided", { code: "invalid_api_key" });
    return undefined;
  }

  const resolved = await services.users.resolveApiKey(bearerToken);
  if (!resolved) {
    sendApiError(reply, 401, "Incorrect API key provided", { code: "invalid_api_key" });
    return undefined;
  }

  return {
    userId: resolved.userId,
    authType: "apiKey",
    scopes: resolved.scopes,
    apiKeyId: resolved.apiKeyId,
  };
}
```

Key points:
- No Supabase JWT session lookup (no `getSessionPrincipal` call)
- No `x-api-key` header fallback
- No dev-user prefix check
- Uses existing `readBearerToken()` (already in scope, line 52)
- Uses existing `sendApiError` import (already imported)
- Two distinct 401 messages: `"No API key provided"` (missing header) vs `"Incorrect API key provided"` (invalid token)
- Both use `code: "invalid_api_key"` and will auto-map to `type: "authentication_error"` via STATUS_TO_TYPE
- Does NOT check `requiredScope` against `resolved.scopes` or call `requirePermission`/`canUse` — scope enforcement is orthogonal and will be handled by the caller or future phases. The `requiredScope` parameter is kept for API compatibility with callers but is unused in the v1 path for now.

DO NOT modify `resolvePrincipal`, `requirePrincipal`, `requireApiPrincipal`, or `requireApiUser` — they serve non-v1 routes.

**In `apps/api/src/routes/v1-plugin.ts`:**

Add an `onSend` hook inside the `v1Plugin` function body, after the `setNotFoundHandler` call and before the route registrations. This hook enforces `Content-Type: application/json; charset=utf-8` on all non-streaming responses:

```typescript
app.addHook('onSend', async (_request, reply, payload) => {
  const ct = reply.getHeader('content-type');
  if (typeof ct === 'string' && ct.includes('text/event-stream')) {
    return payload;
  }
  reply.header('content-type', 'application/json; charset=utf-8');
  return payload;
});
```

Place this after `setNotFoundHandler` and before `registerChatCompletionsRoute`. The hook skips override if the content-type is already `text/event-stream` (streaming responses set their own content-type).
