**A. Add embeddings types to `types.go`.**

Add to the existing file:
- `EmbeddingsRequest` struct -- `Model` string `json:"model"`, `Input` json.RawMessage `json:"input"`, `EncodingFormat` *string `json:"encoding_format,omitempty"`, `Dimensions` *int `json:"dimensions,omitempty"`, `User` *string `json:"user,omitempty"`.

**B. Create `apps/edge-api/internal/inference/embeddings.go`.**

Function `handleEmbeddings(o *Orchestrator, w http.ResponseWriter, r *http.Request)`:

1. Read request body (10MB limit).
2. Parse into `EmbeddingsRequest`. Validate: `Model` required, `Input` required and non-empty.
3. Build `NeedFlags`: `NeedEmbeddings: true`.
4. Estimate credits: embeddings are cheaper, use 1000 credits as default reservation.
5. Call `o.executeSync(...)` with:
   - `endpoint: EndpointEmbeddings`
   - `dispatch: o.litellm.Embeddings`
   - normalize function that:
     a. Unmarshals LiteLLM response into `EmbeddingsResponse`.
     b. Overwrites `Model` with Hive alias.
     c. Ensures `Object` is `"list"`.
     d. Ensures each item has `Object: "embedding"`.
     e. Strips any unknown top-level fields.
     f. Returns cleaned JSON and usage (convert `EmbeddingsUsage` to `UsageResponse` format for the orchestrator: `PromptTokens` maps to `PromptTokens`, `TotalTokens` maps to `TotalTokens`, `CompletionTokens` = 0).

6. **Capability gating for `dimensions` parameter:**
   After route selection in the orchestrator flow, before dispatch, check if `req.Dimensions != nil`. The routing service `SelectionResult` does not currently expose a `SupportsDimensions` flag. For Phase 6, use a pragmatic approach: check if the `dimensions` parameter is present in the request and pass it through to LiteLLM. If LiteLLM / upstream provider returns an error about unsupported dimensions, the existing provider-blind error handler will catch it. For known non-supporting aliases, add a check in the embeddings handler: if `req.Dimensions != nil`, inspect the model alias name. If the alias does not contain `"embedding-3"` or similar known-supporting pattern, write `writeUnsupportedParamError(w, "dimensions", req.Model)`. This is a pragmatic heuristic -- a future phase can add a proper `SupportsDimensions` capability flag to the routing types.

7. **encoding_format handling:** Pass through to LiteLLM. The `encoding_format` field stays in the raw body and LiteLLM handles provider translation. The normalize function preserves `Embedding` as `json.RawMessage` to handle both float arrays and base64 strings without type assertion.
