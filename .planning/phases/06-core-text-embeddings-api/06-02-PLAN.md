**B. Create `apps/edge-api/internal/inference/stream.go`.**

This file implements SSE relay for chat/completions and legacy completions.

Function `(o *Orchestrator) executeStreaming(ctx context.Context, w http.ResponseWriter, r *http.Request, endpoint string, body []byte, model string, aliasID string, needFlags NeedFlags, estimatedCredits int64, includeUsage bool, dispatch func(ctx context.Context, litellmModel string, body []byte) (*http.Response, error)) error`:

Lifecycle:
1. Authorize (same as executeSync).
2. Select route (same).
3. Validate capabilities -- call `validateReasoningCapability` if reasoning is requested.
4. Start attempt with status `"streaming"`.
5. Create reservation.
6. Dispatch to LiteLLM (the body already has `stream: true`).
7. Assert `http.Flusher` on `w`. If not available, release reservation and return 500.
8. Set SSE headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`, `X-Accel-Buffering: no`.
9. `w.WriteHeader(http.StatusOK)` -- commit the response.
10. Create a `UsageAccumulator` struct to track tokens across chunks.
11. Use `bufio.Scanner` on the upstream response body. For each line:
    - If line is `"data: [DONE]"`: write `"data: [DONE]\n\n"`, flush, break.
    - If line starts with `"data: "`: extract the JSON chunk. Parse it into `ChatCompletionChunk` (or generic map for completions). Overwrite `model` field with `aliasID`. Strip unknown fields by re-marshaling from the typed struct. If the chunk contains `usage` (terminal usage chunk), accumulate tokens. Write `"data: {sanitized_json}\n\n"`, flush.
    - If line is empty: write `"\n"`, flush (SSE blank line delimiter).
    - Otherwise: write `"{line}\n"` (pass through event: lines, etc.).
12. **Terminal usage chunk synthesis**: If `includeUsage` was true but the upstream did NOT send a terminal usage chunk (detected by checking if UsageAccumulator has data), synthesize one: `{"id":"...","object":"chat.completion.chunk","created":...,"model":"{aliasID}","choices":[],"usage":{...}}` followed by `data: [DONE]\n\n`.
13. Finalize reservation with accumulated usage. Record completed usage event.
14. **Context cancellation handling**: Use `defer` at the top of the function. If the function exits without finalization (tracked by a `finalized` bool), release reservation with reason `"client_disconnect"` and record an `interrupted` usage event with whatever tokens were accumulated so far. This handles client disconnects mid-stream.

Helper `UsageAccumulator` struct:
- Fields: `InputTokens`, `OutputTokens`, `ReasoningTokens`, `CachedInputTokens`, `CacheReadTokens`, `CacheWriteTokens`, `TotalTokens`, `HasUsage` bool.
- Method `Accumulate(chunk ChatCompletionChunk)` -- if chunk.Usage is non-nil, copy all fields.
- Method `ToUsageResponse() *UsageResponse` -- construct the full `UsageResponse` with `CompletionTokensDetails` and `PromptTokensDetails`.

Function `handleResponses(o *Orchestrator, w http.ResponseWriter, r *http.Request)`:

1. Read and parse `ResponsesRequest`. Validate `Model` (required) and `Input` (required).
2. **Translate to chat/completions request for LiteLLM dispatch:**
   - Map `Input` to `messages`: If `Input` is a string, create `[{"role":"user","content":input_string}]`. If `Input` is an array, pass through as messages (the Responses API message format is close enough to chat/completions for LiteLLM to handle, with minor differences).
   - Map `Instructions` to a system message prepended to messages.
   - Map `Text.format` to `response_format`: if `text.format.type` is `"json_schema"`, translate to `{"type":"json_schema","json_schema":{...}}`.
   - Map `Reasoning.effort` to `reasoning_effort` parameter.
   - Map `MaxOutputTokens` to `max_completion_tokens`.
   - Map `Tools` from Responses API format to chat/completions format: Responses API uses `{type:"function", name:"...", description:"...", parameters:{...}, strict:true}` while chat/completions uses `{type:"function", function:{name:"...", description:"...", parameters:{...}, strict:true}}`. Wrap each tool's fields inside a `function` key.
   - Build a JSON body that is a valid chat/completions request.
3. Build `NeedFlags`: `NeedResponses: true`, `NeedStreaming: req.Stream`, `NeedReasoning: len(req.Reasoning) > 0`.
4. If not streaming: call `o.executeSync(...)` with the translated body and `o.litellm.ChatCompletion` dispatch. The normalize function translates the `ChatCompletionResponse` into a `ResponseObject`:
   - Generate response ID: `"resp_" + uuid`.
   - Set `Object: "response"`, `CreatedAt` from chat completion `Created`, `Model` to Hive alias.
   - Set `Status: "completed"`.
   - Build `Output` array: for each choice, create a `ResponseOutputItem` with `Type: "message"`, `ID: "msg_" + uuid`, `Status: "completed"`, `Role: "assistant"`, `Content` array. If choice has text content, add `ResponseContentPart{Type: "output_text", Text: content, Annotations: []}`. If choice has tool_calls, translate to Responses API function_call output items.
   - Translate `Usage` from chat-completions format to Responses API format: `PromptTokens` -> `InputTokens`, `CompletionTokens` -> `OutputTokens`, `CompletionTokensDetails.ReasoningTokens` -> `OutputTokensDetails.ReasoningTokens`, `PromptTokensDetails.CachedTokens` -> `InputTokensDetails.CachedTokens`.
   - `Store` is always treated as false per PRIV-01 (no transcript storage). Accept the parameter but ignore it.
   - Set `Truncation: "disabled"`, `ToolChoice: "auto"`, `Tools: []` (or from request), `IncompleteDetails: null`, `Error: null`.
5. If streaming: call `o.executeResponsesStreaming(...)`.

**C. Create `apps/edge-api/internal/inference/stream_responses.go`.**

This implements the Responses API lifecycle event translation state machine.

Function `(o *Orchestrator) executeResponsesStreaming(ctx context.Context, w http.ResponseWriter, r *http.Request, body []byte, req ResponsesRequest, aliasID string, needFlags NeedFlags, estimatedCredits int64) error`:

Lifecycle (authorize, route, start attempt, reserve -- same as standard streaming). Then:

1. Dispatch to LiteLLM `/chat/completions` with `stream: true, stream_options: {include_usage: true}` in the translated body.
2. Assert `http.Flusher`. Set SSE headers. Commit response.
3. Build a `responsesEventTranslator` state machine with fields:
   - `responseID` string (generated once: `"resp_" + uuid`)
   - `aliasID` string
   - `created` int64
   - `started` bool -- has `response.created` been emitted?
   - `outputItemAdded` bool -- has first `response.output_item.added` been emitted?
   - `contentPartAdded` bool -- has first `response.content_part.added` been emitted?
   - `outputItems` []ResponseOutputItem -- accumulated items
   - `currentContent` strings.Builder -- accumulated text for current content part
   - `usageAccumulator` UsageAccumulator
   - `finishReason` *string

4. For each upstream SSE line:
   - Parse `data: {json}` into `ChatCompletionChunk`.
   - On first chunk: emit `event: response.created\ndata: {response object with status:"in_progress"}\n\n`. Set `started = true`.
   - On first content delta (chunk.Choices[0].Delta.Content != nil && !outputItemAdded):
     - Emit `event: response.output_item.added\ndata: {"type":"response.output_item.added","output_index":0,"item":{type:"message",id:"msg_...",status:"in_progress",role:"assistant",content:[]}}\n\n`.
     - Emit `event: response.content_part.added\ndata: {"type":"response.content_part.added","output_index":0,"content_index":0,"part":{"type":"output_text","text":"","annotations":[]}}\n\n`.
     - Set `outputItemAdded = true`, `contentPartAdded = true`.
   - On each content delta: emit `event: response.output_text.delta\ndata: {"type":"response.output_text.delta","output_index":0,"content_index":0,"delta":"the text"}\n\n`. Accumulate text.
   - On `finish_reason` set (chunk.Choices[0].FinishReason != nil):
     - Emit `event: response.content_part.done\ndata: {"type":"response.content_part.done","output_index":0,"content_index":0,"part":{"type":"output_text","text":"full accumulated text","annotations":[]}}\n\n`.
     - Emit `event: response.output_item.done\ndata: {"type":"response.output_item.done","output_index":0,"item":{full item with status:"completed"}}\n\n`.
   - On usage chunk (chunk.Usage != nil): accumulate usage.
   - On `data: [DONE]`:
     - Emit `event: response.completed\ndata: {"type":"response.completed","response":{full response object with status:"completed", usage, output, etc.}}\n\n`.
     - Do NOT emit `data: [DONE]` -- the Responses API stream does not use `[DONE]` sentinel.
5. Flush after every event emission.
6. Finalize reservation. Record usage event.
7. Same defer-based cleanup for client disconnects as in `stream.go`.

**Important SSE format difference:** The Responses API uses named events (`event: response.created\ndata: ...\n\n`) while chat/completions uses only `data:` lines. This is critical for SDK compatibility.
