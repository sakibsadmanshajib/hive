    5. Return `RailEvent{ProviderIntentID: pi.ID, EventType: mapped, RawPayload: rawBody}`

  <action>
    **CRITICAL — ProviderIntentID consistency:**
    Each rail's `Initiate` stores a provider-assigned ID via `UpdateProviderDetails(providerIntentID=...)`.
    Each rail's `ProcessEvent` MUST return that same provider-assigned ID as `RailEvent.ProviderIntentID`.
    The service uses `GetPaymentIntentByProviderID` to look up the intent — if ProcessEvent returns a
    different ID (like the Hive UUID), the lookup will fail.

    - Stripe: Initiate stores `pi.ID` (e.g. "pi_xxx"), ProcessEvent returns `pi.ID` from webhook — consistent.
    - bKash: Initiate stores bKash `paymentID` from create response, ProcessEvent MUST return `paymentID` from execute response — NOT `merchantInvoiceNumber`.
    - SSLCommerz: Initiate stores `sessionkey` from initiation response, ProcessEvent MUST return `sessionkey` extracted from IPN body — NOT `tran_id`.

    `type Rail struct` with fields: `httpClient *http.Client`, `baseURL string`, `appKey string`, `appSecret string`, `username string`, `password string`

    `func NewRail(httpClient *http.Client, baseURL, appKey, appSecret, username, password string) *Rail`

    `func (r *Rail) RailName() payments.Rail` — returns `payments.RailBkash`

    `func (r *Rail) grantToken(ctx context.Context) (string, error)`:
    1. POST to `{baseURL}/tokenized/checkout/token/grant`
    2. Headers: `username: r.username`, `password: r.password`, `Content-Type: application/json`
    3. Body: `{"app_key": r.appKey, "app_secret": r.appSecret}`
    4. Parse response: extract `id_token` string
    5. 10-second timeout via context
    6. Return (idToken, nil) or error

    `func (r *Rail) Initiate(ctx context.Context, input payments.InitiateInput) (payments.InitiateResult, error)`:
    1. Call `r.grantToken(ctx)` to get fresh id_token (never cache across sessions)
    2. POST to `{baseURL}/tokenized/checkout/create`
    3. Headers: `Authorization: r.idToken`, `X-App-Key: r.appKey`, `Content-Type: application/json`
    4. Body JSON:
       - `mode`: `"0011"`
       - `payerReference`: `input.AccountID.String()`
       - `callbackURL`: `input.CallbackBaseURL + "/webhooks/bkash/callback"`
       - `amount`: format `input.AmountLocal` as string BDT with 2 decimals: `fmt.Sprintf("%.2f", float64(input.AmountLocal)/100.0)` — NOTE: float64 is acceptable here ONLY for display formatting of an int64 value, not for arithmetic
       - `currency`: `"BDT"`
       - `intent`: `"sale"`
       - `merchantInvoiceNumber`: `input.PaymentIntentID.String()`
    5. Parse response: extract `paymentID` and `bkashURL`
    6. Return `InitiateResult{ProviderIntentID: paymentID, RedirectURL: bkashURL, ExpiresAt: time.Now().Add(10*time.Minute)}`

    `func (r *Rail) ProcessEvent(ctx context.Context, rawBody []byte, headers map[string]string) (payments.RailEvent, error)`:
    1. Parse rawBody as JSON to extract `paymentID` and `status` fields from the bKash callback
    2. Even if status == "success", ALWAYS call Execute endpoint server-side
    3. Grant a fresh token: `r.grantToken(ctx)`
    4. POST to `{baseURL}/tokenized/checkout/execute`
    5. Headers: `Authorization: idToken`, `X-App-Key: r.appKey`
    6. Body: `{"paymentID": paymentID}`
    7. Parse response: check `transactionStatus`
       - `"Completed"` -> EventType = "payment.succeeded"
       - Anything else -> EventType = "payment.failed"
    8. Return `RailEvent{ProviderIntentID: paymentID, EventType: mapped, RawPayload: rawBody}`
       IMPORTANT: ProviderIntentID = bKash `paymentID` (provider-assigned), NOT `merchantInvoiceNumber` (Hive UUID).
       This must match the paymentID stored by Initiate via UpdateProviderDetails so GetPaymentIntentByProviderID can look it up.
