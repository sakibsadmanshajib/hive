# PAY-14-09 — credit_grants append-only at schema level (DB trigger)

## Truth

`credit_grants` is the audit log of every discretionary credit issuance.
Append-only is enforced at the **schema** layer — not just the
application layer — so a malicious or buggy code path cannot mutate or
delete a grant row. Phase 21 chat-app and any future tier-aware RBAC
caller can rely on the row as immutable evidence.

## Implementation

- Migration `20260428_01_budgets_alerts_invoices_grants.sql` installs:
  - Trigger function `tg_credit_grants_immutable` raising the exception
    `append-only ledger violation: credit_grants is immutable`.
  - Row-level trigger `credit_grants_immutable_trg` firing
    `BEFORE UPDATE OR DELETE`.
- The table deliberately omits an `updated_at` column. The Go layer
  (`grants.Repository`) does not expose Update / Delete methods — both
  enforcement layers reinforce one another.
- CHECK constraints:
  - `credit_grants_amount_bdt_subunits_check` (`> 0`).
  - `credit_grants_currency_check` (`= 'BDT'`).

## Tests

- pg-test (Task 2 transcript) — `UPDATE` raises trigger error; `DELETE`
  raises trigger error.
- `TestImmutability_UpdateBlocked` (`grants/audit_test.go`).
- `TestImmutability_DeleteBlocked` (`grants/audit_test.go`).
- `TestRepository_HasNoUpdateOrDeleteSurface` — compile-time guard on
  the Go interface.
