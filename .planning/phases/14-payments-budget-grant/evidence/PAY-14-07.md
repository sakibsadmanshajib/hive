# PAY-14-07 — credit_grants UPDATE/DELETE raises Postgres exception via row trigger

## Truth

The `public.credit_grants` table is append-only at schema level. Any UPDATE or
DELETE attempt — whether application-mediated or direct SQL — raises a
`integrity_constraint_violation` exception with the message
`'credit_grants is append-only — UPDATE/DELETE forbidden (append-only ledger violation)'`.

## Implementation

Migration `20260428_01_budgets_alerts_invoices_grants.sql` lines 105-118
define `tg_credit_grants_immutable()` and the BEFORE UPDATE OR DELETE trigger
`credit_grants_immutable_trg`:

```sql
CREATE OR REPLACE FUNCTION public.tg_credit_grants_immutable()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
    RAISE EXCEPTION 'credit_grants is append-only — UPDATE/DELETE forbidden (append-only ledger violation)'
        USING ERRCODE = 'integrity_constraint_violation';
END;
$$;

CREATE TRIGGER credit_grants_immutable_trg
    BEFORE UPDATE OR DELETE ON public.credit_grants
    FOR EACH ROW EXECUTE FUNCTION public.tg_credit_grants_immutable();
```

## Application-layer mirror

`grants.Repository` interface omits `Update` and `Delete` methods entirely.
Callers cannot accidentally request a mutation that would fail at runtime —
the absence of the API surface is the application-layer mirror of the
schema-level constraint.

## Verification

The trigger was applied as part of Phase 14 Task 2 migration (already
deployed). Task 5 enforces the contract at the application boundary by
omitting Update/Delete from the Repository interface; future PRs cannot
introduce a mutation path without explicitly adding a method that contradicts
the existing schema discipline.
