# PAY-14-02 — Hard-cap enforced on edge-api hot path with 402 + provider-blind body

## Truth

The hard-cap check sits before any upstream provider call. A workspace
that has exhausted its hard cap receives HTTP 402 with a BDT-only,
provider-blind error body — the request never touches OpenRouter / Groq.
This is the regulatory + cost-control invariant; the gate cannot live
on the control-plane round-trip path because hot-path latency forbids it.

## Implementation

- `apps/edge-api/internal/limits/budget_gate.go` — middleware reads
  `budget:hard_cap:{ws}` (Redis) and `budget:mtd_spend:{ws}:YYYY-MM`
  (Redis); both compared with `math/big.Int.Cmp`. Float64 deliberately
  absent.
- Cache miss = fail-open (forward request). Acceptable trade-off: better
  to bill a few extra requests than to tank the hot path.
- 402 body shape:
  `{"error":{"type":"budget_hard_cap_exceeded","code":"BUDGET_HARD_CAP","message":"…","currency":"BDT","amount_bdt_subunits":"…"}}`
  Zero `amount_usd|usd_|fx_|exchange_rate|price_per_credit_usd` keys
  (asserted by test).
- Control-plane PUSHES a fresh `budget:hard_cap:{ws}` value on every
  `SetBudget` / `DeleteBudget` so the gate is read-through with TTL safety.

## Tests

- `TestBudgetGate_AllowsBelowCap` — under-cap request passes to upstream.
- `TestBudgetGate_Returns402AboveCap` — over-cap request receives 402
  with provider-blind body and no upstream call observed.
- `TestBudgetGate_FailsOpenOnRedisError` — gate forwards request when
  Redis is down (logged, not blocked).
- `TestBudgetGate_BodyHasZeroUSDKeys` — regex assertion.
