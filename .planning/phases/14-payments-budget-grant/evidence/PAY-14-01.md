# PAY-14-01 — Workspace soft + hard budget caps (BDT subunits)

## Truth

Each workspace has a single budget row carrying both a soft cap (alert
trigger) and a hard cap (pre-provider 402 block). Both quantities are stored
as BIGINT BDT subunits — the wire shape exposes them as decimal strings so
clients drive `math/big` parsing instead of float64.

## Implementation

- Migration `20260428_01_budgets_alerts_invoices_grants.sql` creates table
  `public.budgets` with columns `soft_cap_bdt_subunits BIGINT`,
  `hard_cap_bdt_subunits BIGINT`, `currency TEXT CHECK (currency = 'BDT')`,
  `(soft_cap, hard_cap)` integrity check (`hard >= soft >= 0`).
- `apps/control-plane/internal/budgets/types.go` — `Budget` struct mirrors
  the table; `BudgetUpsert` payload validated by service layer.
- `apps/control-plane/internal/budgets/http.go` — handles
  `GET/PUT/DELETE /api/v1/budgets/{workspace_id}` and the legacy
  `/api/v1/accounts/current/budget` surface.
- `apps/control-plane/internal/budgets/service.go` — single-source-of-truth
  cache invalidation: every PUT pushes a new `budget:hard_cap:{ws}` value
  into Redis with a 30 s TTL so the edge-api gate (PAY-14-02) sees fresh
  values on the hot path.
- OpenAPI `packages/openai-contract/spec/paths/budgets.yaml` documents the
  full surface; lint-no-customer-usd asserts BDT-only shape.

## Tests

- `TestBudgetService_Upsert_Validates` — rejects `hard < soft`, `soft < 0`.
- `TestBudgetHandler_GetPutDelete_HappyPath` — full CRUD round-trip.
- `TestBudgetService_PushesHardCapToRedisOnUpsert` — gate cache stays fresh.
