# PAY-14-04 — Alert idempotency per (workspace, period, threshold)

## Truth

Re-running the evaluator over the same period MUST NOT produce duplicate
notifications. Idempotency is enforced by stamping
`spend_alerts.last_fired_period` to the period start (`YYYY-MM-01`) and
guarding all subsequent crossings with that timestamp.

## Implementation

- Idempotency key: tuple `(workspace_id, period_start, threshold_pct)`
  encoded by the row PK + `last_fired_period`.
- `budgets.CronEvaluator` updates `last_fired_period` ONLY after the
  notifier returns success — failed dispatch leaves the row unchanged so
  the next pass retries.
- Period rollover: `last_fired_period` is compared against the current
  month start; new month resets the eligible-to-fire condition without an
  explicit cron sweep.

## Tests

- `TestEvaluator_OneShotPerPeriodPerThreshold` — first pass fires once,
  second pass within the same period fires zero.
- `TestEvaluator_NotifierFailureDoesNotStamp` — failed dispatch must be
  retried on the next pass (notifier observed twice).
- `TestEvaluator_NewMonthReFires` — period rollover re-arms the alert.
