# v1.1 Deferred Scope

## What v1.0 Ships

Phases 1–10. OpenAI-compatible developer API:

- Text: chat/completions (streaming + non-streaming), completions, responses, embeddings.
- Media: images (generation/edits), audio (speech/transcription/translation), files, uploads.
- Async: batches — create, attribution, failure-path terminal settlement.
- Billing: Stripe, bKash, SSLCommerz. BDT top-ups with FX + 3% fee.
- Infra: Supabase Storage (S3), LiteLLM routing (OpenRouter + Groq), Redis, Postgres.
- Ops: provider-blind errors, prepaid reservations (strict + temporary overage), cold-start healthcheck windows.

Fit for: chat applications, CLI coding agents, drop-in replacements of OpenAI SDK base URL + key.

### New Gaps Surfaced During v1.0 Stabilization

5. **Batch success-path terminal settlement**
   - Source: `.planning/phases/10-routing-storage-critical-fixes/KNOWN-ISSUE-batch-upstream.md`
   - Current: failure-path settlement works; success-path blocked because LiteLLM's managed file upload does not support openrouter/groq.
   - Recommended: local batch executor in control-plane (fan-out `/v1/chat/completions`, compose output/error JSONL, settle from per-request usage). Decision deferred to phase planning.
6. **`ensureCapabilityColumns` wrong-table fix**
   - File: `apps/control-plane/internal/routing/repository.go`
   - Current: targets `route_capabilities` instead of `provider_capabilities`. Latent bug documented in CLAUDE.md Known Issues. Routing currently works because separate seed path populates needed columns, but the capability DDL path is dead code.
7. **`amount_usd` on BD checkout (regulatory)**
   - File: `apps/control-plane/internal/payments/http.go:105-115`
   - Current: BD-visible payment response includes `amount_usd`. Contravenes BD regulatory rule (no FX display). Deferred — v1.0 BD checkout treated as preview, not public launch.
8. **RBAC + verification-aware authorization model**
   - Files: `apps/control-plane/internal/accounts/types.go`, `apps/control-plane/internal/accounts/service.go`, `apps/web-console/lib/viewer-gates.ts`
   - Current: authorization is limited to workspace membership roles (`owner` / `member`) plus feature-specific booleans like `CanInviteMembers` and `CanManageAPIKeys`. That is not a reusable RBAC layer and it does not cover guest or unverified-user scopes consistently.
   - Recommended: design a shared permission matrix and enforce it in both control-plane handlers and web-console route gating so unverified or limited users cannot reach billing, API-key, analytics, or other sensitive surfaces by accident.
